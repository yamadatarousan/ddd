# DDD学習メモ

## このドキュメントの読み方
この文書は、次の順序で理解できるように構成している。
1. DDDの目的（なぜDDDを使うのか）
2. 先に知るべき基本用語（モデル、エンティティ、値オブジェクトなど）
3. このアプリに当てはめるとどうなっているか
4. まだ未対応の課題は何か

## 1. そもそもDDDとは何か
Domain-Driven Design（ドメイン駆動設計）は、
「業務の本質（ドメイン）を中心にモデルを作り、そのモデルを実装と会話の共通土台にする設計アプローチ」です。

重要なのは次の2点。
- コードを業務ルールの表現に寄せること
- 開発者と業務側が同じ言葉でモデルを育てること

## 2. 先に押さえる基本用語
### 2.1 モデル
DDDでいう「モデル」は、単なるデータ構造ではない。
業務の概念・ルール・関係性を、チームが同じ理解で扱える形にしたもの。

モデルは「テーブル定義」や「JSON項目一覧」ではなく、
「この業務で何が正しい状態か」を表現するための地図。

### 2.2 ドメインモデル
ドメインモデルは、モデルの中でも「業務ルールを表す中心部分」。
アプリの都合（画面都合、DB都合）ではなく、業務都合で設計する。

### 2.3 ユビキタス言語（Ubiquitous Language）
チームで統一して使う業務用語。
会話で使う言葉と、コードの名前をそろえることで仕様と実装のズレを減らす。

### 2.4 エンティティ（Entity）
IDで同一性を持つオブジェクト。
例: `Todo`（タイトルが変わっても同じIDなら同じTodo）。

必要性:
- 業務上「同じものを追跡したい」対象を扱うため。
- `Todo` はタイトルや状態が変わっても、同じIDなら同一の仕事として扱う必要がある。
- これを値比較だけで扱うと、更新のたびに別物として誤認しやすい。

このアプリでの効果:
- `todo-1` は `ChangeTitle` や `Complete/Reopen` 後も同じ `Todo` として扱える。
- 更新・削除の対象をIDで一意に決定できる。

### 2.5 値オブジェクト（Value Object）
値そのものに意味を持つ不変オブジェクト。
例: `Title`（空文字不可、文字数制約あり）。

必要性:
- 「ただの文字列/数値」に業務ルールを散らさないため。
- 値に意味と制約を持たせることで、不正値の流入を早い段階で防げる。
- ルールが1か所に集まるので、変更時の影響範囲を小さくできる。

このアプリでの効果:
- `Title` 生成時に空文字・文字数超過を必ず検証できる。
- 画面・HTTP・ユースケースで同じ検証を重複実装しなくて済む。

注意点:
- `Todo` と `Title` は同列ではない。
- `Todo` はエンティティ。
- `Title` は `Todo` を構成する値オブジェクト。

### 2.6 集約（Aggregate）/集約ルート（Aggregate Root）
集約は「整合性を守りながら、1回の変更として扱う境界」。
集約ルートは、その集約への唯一の入口になるエンティティ。

先に結論:
- 集約は「どこまでを一緒に正しく保つか」を決めるための概念
- 集約ルートは「その境界の責任者」

必要性:
- 関連オブジェクトを無制限に直接更新できると、整合性が壊れやすい
- 更新ルールを1か所に集めないと、同じ業務ルールが各所でバラバラに実装される
- 排他制御やトランザクションの単位を決めにくくなる

基本ルール:
1. 外部からは集約ルート経由でしか状態変更しない
2. 集約内の不変条件（常に守るルール）は集約ルートが守る
3. リポジトリは基本的に集約ルート単位で扱う
4. 1つのトランザクションで強く整合性を保証する範囲を集約にする

このアプリでの位置づけ:
- 現在のTodo管理では、`Todo` 単体で整合性が完結している
- そのため `Todo` が集約ルート相当
- `Complete`, `Reopen`, `ChangeTitle` は `Todo` を通して状態遷移する

今後の拡張で集約設計が重要になる例:
- `ChecklistItem` を `Todo` 配下に持つ場合
- 「Todoが完了なら全ChecklistItemも完了でなければならない」などのルールを追加する場合

`ChecklistItem` とは（このドキュメント上の仮の例）:
- Todoをさらに細分化した作業項目
- 例: `買い物Todo` に対して `牛乳`, `パン`, `卵` のような子タスク
- 現在の実装にはまだ存在しない概念で、将来拡張を説明するために使っている

この場合の考え方:
- `Todo` を集約ルートにして、`ChecklistItem` の更新も `Todo` 経由で行う
- ルートを通すことで、ルール違反の更新経路を減らせる

### 2.7 アプリケーションサービス（Use Case）
アプリケーションサービス（このプロジェクトではUse Case）は、  
「利用者がやりたい操作」を、ドメインモデルを使って実行する調停役。

重要なのは「手続きの順序と境界」を管理する点で、役割は次の通り。
1. 入力（コマンド）を受け取り、最低限の形式確認をする
2. 必要なドメインオブジェクトを取得する（Repository経由）
3. ドメインの振る舞いを呼び出す（`Complete`, `Reopen`, `ChangeTitle` など）
4. 結果を保存する（Repository経由）
5. 呼び出し元（HTTP）へ結果またはエラーを返す

必要性:
- HTTP層に業務手順を混ぜないため
- ドメインオブジェクトに「永続化や取得の都合」を持ち込まないため
- 1つのユースケースの処理順序を1か所で追えるようにするため

「やること」と「やらないこと」:
- やること: 手順のオーケストレーション（調停）
- やらないこと: 業務ルールそのものの実装（それは `domain` の責務）
- やらないこと: HTTPレスポンス形式の決定（それは `interfaces/http` の責務）
- やらないこと: 具体的なDB操作の詳細（それは `infrastructure` の責務）

このアプリでの具体例:
- `CreateTodoUseCase`: タイトル検証済みの `Todo` を作成して保存
- `CompleteTodoUseCase`: 対象取得 → `entity.Complete()` → 保存
- `ReopenTodoUseCase`: 対象取得 → `entity.Reopen()` → 保存
- `UpdateTodoTitleUseCase`: 対象取得 → `entity.ChangeTitle(value)` → 保存
- `DeleteTodoUseCase`: ID指定で削除
- `ListTodoUseCase`: 条件付き一覧取得

補足（誤解しやすい点）:
- 「手続きだけ」と言うと薄く見えるが、実際には
  「どのドメイン操作を、どの順序で、どの境界で実行するか」を定義する重要な層。
- ここが曖昧だと、同じユースケース実装がHTTPやUIに分散して崩れやすい。

### 2.8 リポジトリ（Repository）
ドメインオブジェクトの永続化抽象。
ユースケースは保存方式を知らず、インターフェース越しに扱う。

### 2.9 境界づけられたコンテキスト（Bounded Context）
あるモデルと言葉が「そのまま通用する範囲」を区切ったもの。
同じ単語でも、コンテキストが変わると意味が変わるため、境界を切る。

例: `ユーザー` という単語の意味
- Todo管理コンテキスト: Todoを操作する主体
- 請求コンテキスト: 請求先としての顧客
- 通知コンテキスト: 通知の受信者

境界を切らないと起きる問題:
- 1つの変更が無関係な機能へ波及する
- 用語の意味がチーム内でズレる
- モデルが肥大化して理解しづらくなる

## 3. このアプリに当てはめるとどうなっているか
### 3.1 ドメインモデル
- エンティティ: `Todo`
- 値オブジェクト: `Title`
- 主な振る舞い: `Complete`, `Reopen`, `ChangeTitle`

### 3.2 レイヤー構成
- `backend/domain`
  - エンティティと値オブジェクト、不変条件を定義
- `backend/application`
  - ユースケース単位の操作を定義
  - リポジトリはインターフェースに依存
- `backend/infrastructure`
  - リポジトリ実装（現在はインメモリ）
- `backend/interfaces/http`
  - HTTP入力をユースケースコマンドへ変換
  - ユースケース結果をレスポンスへ変換

### 3.3 依存方向
`interfaces/http -> application -> domain`
`infrastructure -> domain`

外側（HTTP、保存方式）の都合が、内側（ドメイン）に入り込みにくい構造になっている。

### 3.4 このアプリでのコンテキスト適用
境界づけられたコンテキスト自体の説明は `2.9` を参照。
ここでは、このアプリでどう適用しているかだけを書く。

このアプリでは、次の2コンテキストを実装している。
- Todo管理コンテキスト: `Todo`, `Title`, 作成/完了/再開/更新/削除
- 通知コンテキスト: `Notification`, 完了通知の記録/一覧

コード上の境界表現（どこで分けているか）:
- Todo管理コンテキスト
  - `backend/domain/todo`
  - `backend/application/todo`
  - `backend/infrastructure/todo/memory`（Todo用Repository）
- 通知コンテキスト
  - `backend/domain/notification`
  - `backend/application/notification`
  - `backend/infrastructure/notification/memory`
- コンテキスト間の橋渡し
  - `backend/integration/todo_notification`

コンテキストを跨いだ連携をどう実現しているか:
1. Todo管理側の完了ユースケースは、外部連携用ポート `TodoCompletionNotifier` を持つ
2. 完了時、Todo管理側は「完了した事実（todoID, title）」だけをポートへ渡す
3. `integration/todo_notification.Notifier` がその入力を通知コンテキストのコマンドへ変換
4. 通知コンテキストの `RecordTodoCompletedUseCase` が `Notification` を生成して保存
5. HTTP層は `GET /notifications` で通知コンテキストの結果を返す

重要な点:
- Todo管理は `Notification` 型を直接知らない
- 通知コンテキストも `Todo` 型を直接知らない
- 共有しているのは「完了事実を渡すためのポート契約」だけ

該当実装:
- ポート: `backend/application/todo/complete_todo.go`
- アダプター: `backend/integration/todo_notification/notifier.go`
- 通知ユースケース: `backend/application/notification/record_todo_completed.go`
- HTTP公開: `backend/interfaces/http/router.go` の `GET /notifications`

## 4. なぜDDDに則っていると言えるか
- 業務ルールが `domain` に集中している
- ユースケースが `application` で明確に分離されている
- リポジトリをインターフェース化し、保存方式へ直接依存していない
- レイヤー依存方向が内側を守る形になっている
- 受け入れテストで、ユースケース連携の一貫性を確認している

## 5. DDD観点で、まだ考慮が足りない点
学習用としては十分進んでいるが、実務レベルでは以下が未整備。

1. コンテキストマップ（関係図）の明文化不足
2. 集約ルール（複数エンティティ整合性）の不足
3. ドメインイベント未導入
4. 永続化モデルとドメインモデルの差分吸収未整備
5. トランザクション境界・同時実行制御の設計不足
6. ユビキタス言語の運用ルール不足
7. 認証・認可が未導入
8. 監査ログ・履歴管理未導入
9. ドメイン観点のエラー分類強化余地

## 6. この学習アプリの到達目標
- ドメインルールを中心に置く設計感覚を掴む
- ユースケース分割と依存方向の意味を理解する
- 境界づけられたコンテキストを意識して拡張点を判断する

実務向けの完全版DDDではなく、
「DDDの核心を壊さない最小構成」を維持することをこのプロジェクトの優先方針とする。
